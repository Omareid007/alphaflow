Critical Issues (P0)

Critical C-001

Position Tracking Duality

Positions are tracked in both the local PostgreSQL database and Alpaca's API. The database positions can become stale, leading to incorrect P&L calculations or duplicate trade attempts. No automatic reconciliation beyond a stub placeholder.

Impact Incorrect trades, wrong P&L
Location server/storage.ts vs alpaca.getPositions()
Effort Medium (1-2 days)
Critical C-002

No Unified Data Orchestration

The TradingCoordinator class exists but only logs events without controlling flow. Multiple components (strategies, trading engine, orchestrator) can make independent decisions without coordination, potentially causing conflicting actions.

Impact Conflicting decisions
Location server/orchestration/coordinator.ts
Effort Medium (2-3 days)

Technical Indicator Redundancy

calculateSMA, calculateRSI, calculateROC functions are duplicated across 4 files (~280 lines). A canonical library exists at server/lib/technical-indicators.ts but strategies implement their own versions.

Triple Sentiment Source Without Orchestration

Sentiment data is fetched from GDELT (free), NewsAPI (80/day), and HuggingFace FinBERT without prioritization or deduplication. Wastes API budget and produces inconsistent results.

Files gdelt.ts, newsapi.ts, huggingface.ts
Effort Medium (1-2 days)

Add Walk-Forward Backtesting Prompt #6
Prompt
OBJECTIVE: Add walk-forward validation to the backtest engine to detect overfitting. CREATE NEW FILE: /server/strategies/backtest/walk-forward-engine.ts INTERFACE: export interface WalkForwardConfig { inSampleRatio: number; // 0.7 = 70% for training numFolds: number; // Number of walk-forward folds optimizationMetric: "sharpe" | "sortino" | "totalReturn"; } export interface WalkForwardResult { inSampleMetrics: BacktestMetrics; outOfSampleMetrics: BacktestMetrics; degradationRatio: number; // OOS/IS performance ratio probabilityOfOverfitting: number; recommendation: "reliable" | "warning" | "overfitted"; } IMPLEMENTATION: 1. Split historical data into IS/OOS periods based on inSampleRatio 2. Run backtest on IS period 3. Run backtest on OOS period with same parameters 4. Calculate degradation ratio = OOS_metric / IS_metric 5. If degradation < 0.5, recommendation = "overfitted" 6. If degradation 0.5-0.8, recommendation = "warning" 7. If degradation > 0.8, recommendation = "reliable" API ENDPOINT: POST /api/strategies/:type/walk-forward-backtest
